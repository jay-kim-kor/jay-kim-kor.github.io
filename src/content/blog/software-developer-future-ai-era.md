## 이 글을 쓰게 된 이유

Mitchell Hashimoto(HashiCorp 공동창업자, Terraform/Vagrant 개발자)의 AI 활용 방법론을 살펴보다가, 한 가지 의문이 생겼습니다. 그의 방식은 여전히 **"AI가 쓴 코드를 사람이 리뷰한다"**는 전제 위에 서 있는데, AI의 방대한 코드 생산량을 사람이 정말 전부 리뷰할 수 있는 걸까?

이 질문을 따라가다 보니, 결국 더 근본적인 질문에 도달했습니다.

> **소프트웨어 개발자라는 직업은 앞으로 어떻게 되는 걸까?**

듄(Dune)에서 폴 아트레이데스는 "과거의 인과를 정밀하게 이해해야 미래를 볼 수 있다"고 했습니다. 그 방식을 빌려, 과거 산업 혁명의 패턴에서 개발자의 미래를 추적해보겠습니다.

---

## Hashimoto 방식: 현시점 최선의 절충안

먼저 Mitchell Hashimoto가 어떻게 AI를 활용하는지 정리합니다. 그는 2026년 2월 블로그에 "My AI Adoption Journey"를 발표하며 6단계 프레임워크를 공개했습니다.

**6단계 요약:**

1. **채팅봇 버리기** — 웹 채팅이 아닌 파일 읽기/실행이 가능한 에이전트로 전환
2. **자기 작업 재현하기** — 먼저 직접 코딩, 그다음 AI에게 같은 작업 시키고 비교
3. **퇴근 시간 에이전트** — 하루 마지막 30분에 에이전트에게 작업을 맡기고 퇴근
4. **확실한 작업 아웃소싱** — 성공률 높은 작업을 AI에 위임, 본인은 다른 일
5. **하네스 엔지니어링** — `AGENTS.md` + 스크립트로 에이전트 실패 패턴을 체계적 제거
6. **항상 에이전트 실행** — "지금 에이전트가 나를 위해 뭘 할 수 있지?" 상시 자문

핵심은 **세션 분할**입니다. 대형 기능을 한 번에 요청하지 않고, 커밋 단위로 쪼갭니다. 실제로 Ghostty macOS 스플릿 리팩토링 PR(+1,641/-1,125)을 16개 세션으로 나눠 진행했고, 80%를 AI가 작성했으며, 비용은 $15.98이었습니다.

그리고 그의 가장 중요한 원칙은 이겁니다:

> **"이해 못 하면 롤백. 최종 수동 검토 없이 AI 코드를 절대 출시하지 말 것."**

## 하지만, 리뷰가 정말 가능한가?

여기서 문제가 생깁니다.

Hashimoto가 +1,641줄 PR을 리뷰할 수 있었던 건, 16개 커밋으로 쪼개서 각각 100줄 단위로 봤기 때문입니다. 그리고 그는 **Ghostty의 창시자**입니다. 코드베이스를 완벽히 이해하는 사람이죠.

하지만 현실의 대부분의 개발자는:

- 자기가 만들지 않은 코드베이스에서 일하고
- AI가 500줄 이상 생성한 코드를 라인 바이 라인으로 보는 건 비현실적이며
- 실제로 하는 건 "대충 훑어보기 → 테스트 통과 확인 → 머지"

이건 리뷰가 아니라 **승인 도장**입니다.

결국 업계가 향하는 방향은, **"리뷰를 잘 하자"**가 아니라 **"리뷰 없이도 안전한 구조를 만들자"**입니다. 테스트 커버리지, 타입 시스템, 형식 검증 같은 것들이 "사람의 눈"을 대체하는 거죠.

그리고 이 흐름은, 소프트웨어 개발자의 역할 자체가 바뀌고 있다는 더 큰 이야기로 이어집니다.

## Next Michigan: 산업은 살고, 노동자는 죽는다

Ian Park의 글 "Next Michigan"은 단순한 주장을 합니다:

> **"노동의 종말은 없다. 하지만 당신의 자리는 없다."**

Michigan/Detroit 비유는 이런 뜻입니다. 자동차 산업은 사라지지 않았습니다. 오히려 성장했습니다. 하지만 **Detroit의 자동차 노동자**는 사라졌습니다.

**Detroit 타임라인:**

- **1950년**: 자동차 고용 정점 (22만 명)
- **1953년**: 자동화 도입 시작 — "아직은 괜찮다"
- **1958년**: 첫 대량 실직 (25만 명) — 불과 5년 만
- **1963년**: 제조업 일자리 15만 개가 교외로 이전
- **1970년**: 고용 절반 이하로 축소
- **1978-82년**: 추가 40만 명 해고 (30% 감소)
- **2009년**: GM 파산

정점에서 첫 대량 해고까지 **5년**. 절반 축소까지 **20년**. 완전 붕괴까지 **60년**.

그리고 핵심은 — 자동차 회사들은 변화에 **느리게 대응**했습니다. 해외 경쟁사를 무시했고, 연비 기술 전환을 거부했습니다. "우리는 괜찮다"는 믿음이 붕괴를 가속화했습니다.

## 과거의 인과: 기술 혁명은 항상 같은 패턴을 따른다

듄의 프레스코트(예지) 방식으로, 과거의 패턴을 추적합니다.

### 패턴 1: 4단계 사이클

모든 기술 혁명은 동일한 사이클을 거쳤습니다:

```
Phase 1: 도구 등장 → "신기하지만 장난감"
Phase 2: 생산성 폭발 → 소수의 얼리어답터가 10x 달성
Phase 3: 임금-생산성 괴리 (Engels' Pause)
         → 생산성은 올라가지만 노동자 임금은 정체
Phase 4: 새로운 균형 → 완전히 다른 직업 구조로 재편
```

1차 산업혁명 때 영국 방직 노동자의 실질 임금은 **40년간 정체**했습니다(1790-1840). 생산성은 폭발적으로 증가했지만, 그 이익은 노동자가 아닌 자본가에게 갔습니다.

**현재 소프트웨어 산업은 Phase 2에서 Phase 3으로 진입 중입니다.**

### 패턴 2: 자동화되는 것의 "한 단계 위"가 새 가치가 된다

| 시대 | 자동화된 것 | 새로운 가치 |
|------|-----------|------------|
| 인쇄기 (1440) | 필사 → 복제 | 편집, 큐레이션 |
| 방직기 (1760) | 수동 직조 → 기계 생산 | 기계 설계, 공장 관리 |
| 자동차 (1910) | 마차 → 엔진 이동 | 도로 설계, 물류 시스템 |
| PC (1980) | 수동 계산 → 자동 계산 | 로직 설계 (프로그래밍) |
| AI (2024~) | 코드 작성 → 자동 생성 | **??? → "무엇을 왜 만들 것인가"** |

**인과 법칙**: 코딩이 자동화되면, 그 한 단계 위 추상화 — 즉 **"무엇을 왜 만들 것인가"를 판단하는 능력** — 가 새로운 희소 자원이 됩니다.

### 패턴 3: 전환 속도는 점점 빨라진다

- 인쇄기: 유럽 전역 확산에 **100년**
- 산업혁명: 영국 → 세계 확산에 **60년**
- PC/인터넷: 보편화까지 **20년**
- 스마트폰: 보편화까지 **10년**
- AI 코딩 도구: 개발자 84% 사용까지 **3년** (2022-2025)

## 미래 예측: 소프트웨어 개발자 타임라인

Detroit 패턴에 현재 데이터를 오버레이합니다.

### 이미 현실인 것 (2020-2025)

- **2020-22년**: 소프트웨어 고용 정점 (팬데믹 채용 붐)
- **2023년**: 빅테크 대량 해고 시작 (30만+ 감원)
- **2024-25년**: AI 도구 보편화 (개발자 84% 사용), 주니어 채용 20% 감소 (스탠포드 연구)

### 예측 구간 (2026-2033)

- **2026년** — Phase 3 본격 진입
  - Anthropic CEO: "AI가 코딩의 90% 처리"
  - Claude Code 창시자 Boris Cherny: "연말까지 '소프트웨어 엔지니어' 타이틀이 사라지기 시작"
  - 주니어 개발자 역할 대폭 축소

- **2027-28년** — Engels' Pause (임금 정체기)
  - 생산성은 3-5x 증가하지만
  - 소프트웨어 인력 수요 감소
  - 임금 하방 압력 본격화

- **2029-30년** — 새로운 직업 구조의 윤곽
  - "코더"는 사실상 소멸
  - "시스템 아키텍트 + AI 오케스트레이터"로 분화
  - 도메인 전문가 + 기술 이해 하이브리드 부상

- **2033년+** — 새로운 균형점
  - 소프트웨어 산업 자체는 더 커짐
  - 하지만 종사자 수는 현재의 30-50%
  - 1인 생산성은 현재의 10-50x

## 선택지: 피봇할 것인가, 버틸 것인가

### 방향 A: 피봇하는 경우

소프트웨어 개발 역량을 **지렛대**로, 다른 영역의 가치를 만드는 방향입니다.

**1. 도메인 + 기술 하이브리드**

코딩은 "도구"로 격하되고, 도메인 지식이 본체가 됩니다. 금융을 아는 개발자, 의료를 아는 개발자, 제조를 아는 개발자. "이 산업을 아는 사람이 AI를 쓴다"는 포지션입니다.

**2. AI 오케스트레이터 / 시스템 인티그레이터**

개별 코드를 작성하는 게 아니라, AI 에이전트 시스템을 설계하고 조율하는 역할. Hashimoto가 말한 "건축가"의 진화형입니다.

**3. 1인 기업 / 기술 기반 창업**

AI로 혼자서 팀 규모의 제품을 구축합니다. 코딩은 AI, 판단/마케팅/고객관계는 인간. "5명이 하던 걸 1명이 한다"는 모델입니다.

### 방향 B: 피봇 없이 살아남는 경우

소프트웨어 개발 역량 자체로 시장에서 오래 버티는 전략입니다.

**1. "마지막 1마일" 전문가**

AI가 90% 코딩해도, 프로덕션 배포/운영/장애대응은 현실 세계의 복잡성과 엮여 있어 자동화가 가장 늦습니다. SRE, 플랫폼 엔지니어링, 보안 — 이 영역은 물리적 인프라와 맞닿아 있어서 순수 코드 생성으로는 대체 불가능합니다.

**2. "검증 시스템 설계자"**

AI 코드를 사람이 전부 리뷰하는 건 이미 불가능합니다. 그래서 "리뷰 없이도 안전한 구조"를 만드는 사람이 필요합니다. 타입 시스템, 형식 검증, 계약 기반 프로그래밍, property-based testing — 이것들이 사람의 눈을 대체하는 구조를 설계하는 역할입니다.

**3. "레거시-AI 브릿지"**

세상의 80%는 레거시 시스템입니다. COBOL, 온프레미스, 규제 산업의 복잡한 시스템을 AI 시대로 이전하는 전문가는 10-20년간 수요가 보장됩니다. AI가 가장 못하는 일 중 하나가 "문서화되지 않은 20년 된 시스템의 실제 동작을 파악하는 것"입니다.

**4. "AI가 못하는 개발"**

하드웨어 인접(임베디드, FPGA, 커널), 극한 성능(HFT, 게임엔진, DB 내부), 규제 필수 인증(의료기기, 항공, 원자력). 학습 데이터 부족 + 실수 비용 극대 = AI 진입이 가장 느린 영역입니다.

## 마무리: 코딩이 아니라, 판단이 남는다

역사가 말하는 인과는 명확합니다.

- 산업은 죽지 않습니다. **역할이 죽습니다.**
- 자동화되는 것의 한 단계 위가 새로운 가치가 됩니다.
- 전환기에는 임금이 정체되고, 적응하지 못한 사람이 도태됩니다.

소프트웨어 산업은 Detroit의 자동차 산업처럼 더 커질 겁니다. 하지만 "코드를 타이핑하는 행위"로 먹고사는 시대는 끝나가고 있습니다.

남는 것은 **판단**입니다. 무엇을 만들 것인가, 왜 만들 것인가, 이것이 정말 작동하는가. 코드를 쓸 줄 아는 사람이 이 판단까지 할 수 있다면 — 그것이 피봇이든 아니든 — 가장 강력한 생존 전략입니다.

Detroit에서 살아남은 건 자동차를 조립하는 사람이 아니라, 자동차의 미래를 설계하는 사람이었습니다.
